# Automated Release Process Guide

This document outlines how the CI/CD pipeline calculates release versions, generates release notes, and builds/publishes Docker images. It also provides troubleshooting steps for correcting or recreating a release.

---

## 1. Versioning Strategy

The CI/CD pipeline creates new version tag automatically when a pull request is merged into the `main` branch.

### Step A: Finding the Current Version
* The action inspects existing Git tags on the remote repository (e.g., `v1.0.0`) using the GitHub REST API to find the highest semantic version.

### Step B: Calculating the Next Version
* It scans the commit messages merged into `main` since the last tag.
* It uses **Conventional Commits** formatting to decide the bump type:
  * **Patch Bump** (`v1.0.0` -> `v1.0.1`): Triggered by commit prefixes starting with `fix:` (e.g., `fix: resolve config load timeout`).
  * **Minor Bump** (`v1.0.0` -> `v1.1.0`): Triggered by commit prefixes starting with `feat:` (e.g., `feat: add support for new DNS provider`).
  * **Major Bump** (`v1.0.0` -> `v2.0.0`): Triggered by commit prefixes containing `BREAKING CHANGE:` or an exclamation mark next to the prefix (e.g., `feat!: rewrite docker client integration`).
  * **Default Fallback:** If no commits match conventional prefixes, the action defaults to a **Patch Bump**.

---

## 2. Release Description & Release Notes

* **Changelog:** A summarized change log is automatically compiled from commit messages since the last tag.
* **GitHub Release Notes:** The release description automatically includes a list of all commit messages, pull requests, and contributors since the previous version, natively generated by GitHub.

---

## 3. Image Publishing Registry

* **GitHub Container Registry (GHCR):** The image is automatically built and pushed to `ghcr.io/techtricks-ai/docker-external-dns` during the main CI/CD pipeline run.
* **Docker Hub:** The pipeline also triggers an additional workflow which runs in isolation, builds the image, and pushes it to Docker Hub (`techtricks/docker-external-dns`).

---

## 4. Cleanly Recreate a Release Tag

If an existing release version ever need to be recreated or overwriten (for example, if a release failed mid-build, or if you deleted and want to republish `v0.0.2` instead of generating `v0.0.3`):

### Releases vs. Git Tags
When you delete a Git tag, GitHub **does not** automatically delete the associated Release object. The release remains in GitHub's database pointing to a commit, which causes the tagging action to still detect the old version and bump past it.

### Steps to Recreate a Version
To cleanly reset and recreate a version:

1. **Delete the GitHub Release & Tag (Via Web UI):**
   * Navigate to **Releases**
   * Click the release you want to recreate (e.g., `Release v0.0.2`) and **Delete** it.
   * Navigate to **Tags**
   * Select the tag you want to recreate (e.g., `v0.0.2`) and **Delete** it.

2. **Trigger the Rebuild:**
   * Trigger the pipeline by **rerunning the job** or **pushing a new commit** to the `main` branch.
